# 核心思想：封装变化
设计模式都是在做一件事：将 **变** 和 **不变** 分离开来。  

* 不变的部分：系统的骨架、逻辑  
* 变的部分：业务规则、算法实现、依赖的第三方接口

比如：一个电商系统需要支付功能。  
没有设计模式时，支付功能可能这样写：alipay.pay()、wechat.pay()、unionpay.pay()。  
有了设计模式后，支付功能可能这样写：定义一个支付接口，然后实现不同的支付方式。
订单服务只认接口，不认具体实现。  
这个就叫 **封装变化**，支付方式在变，但订单处理流程不变。

# 六大设计原则
记忆口诀：**开闭原则、依赖倒置、接口隔离、单一职责、里氏替换、迪米特法则**
1. 单一职责原则（SRP，Single Responsibility Principle）：一个类只负责一件事
2. 里氏替换原则（LSP，Liskov Substitution Principle）：子类可以扩展父类的功能，但不能改变父类原有的功能
3. 依赖倒置原则（DIP，Dependency Inversion Principle）：面向接口编程，依赖于抽象而不依赖于具体
4. 接口隔离原则（ISP，Interface Segregation Principle）：使用多个隔离的接口，比使用单个接口要好
5. 迪米特法则（LoD，Law of Demeter）：一个对象应该对其他对象有最少的了解
6. 开放封闭原则（OCP，Open Closed Principle）：对扩展开放，对修改关闭

# 装饰器模式
装饰器模式（Decorator Pattern）允许向一个现有的对象添加新的功能，
同时又不改变其结构。这种类型的设计模式属于结构型模式，它是作为现有的类的一个包装。

和被装饰的对象实现同一个接口，并持有被装饰对象的引用，
在调用被装饰对象的方法前或后，增加一些额外的功能。  

装饰器模式和代理模式的区别：
* 装饰器模式：扩展功能，原有功能不变且可直接使用
* 代理模式：控制访问，原有功能不变且可直接使用
* 装饰器模式：可以有多个装饰器，装饰器之间可以嵌套
* 代理模式：只能有一个代理类，代理类和被代理类实现同一个接口
* 装饰器模式：装饰器和被装饰对象是同一个类型
* 代理模式：代理类和被代理类是同一个类型
* 装饰器模式：装饰器和被装饰对象是同一个对象

特别适合需要叠加组合功能的场景，比如：
* 日志记录
* 权限控制
